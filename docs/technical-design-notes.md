# 技术设计笔记

## 分层架构

后端采用 Spring Boot 常见分层结构：

- Controller：负责 HTTP 请求参数接收、角色权限注解和响应 DTO 返回。
- Service：负责业务规则、事务边界、状态流转和跨模块协作。
- Repository：负责 JPA 数据访问。
- Domain：负责核心实体和枚举建模。
- DTO：负责接口输入输出结构隔离。

这种结构让订单、支付、风险、缓存和审计模块各自保持清晰边界，便于后续替换数据库、缓存和认证实现。

## 订单状态机

订单创建并不直接代表交易完成，而是先进入 `PENDING_PAYMENT` 并锁定库存。支付成功后订单变为 `PAID`，待支付订单可以关闭为 `CLOSED`，已支付订单可以退票为 `REFUNDED`。

允许流转：

```text
PENDING_PAYMENT -> PAID
PENDING_PAYMENT -> CLOSED
PAID -> REFUNDED
```

禁止流转：

- 待支付订单不能退票。
- 已支付订单不能关闭。
- 已关闭订单不能支付、关闭或退票。
- 已退票订单不能重复退票。

## 库存一致性

下单时涉及库存扣减和订单创建，关闭或退票时涉及订单状态更新和库存释放。这些操作放在事务中完成，保证库存和订单状态保持一致。

库存表使用 `version` 字段开启 JPA 乐观锁。多个请求同时锁定同一条库存时，只有先提交的事务能成功更新版本号，后提交事务会触发冲突并返回明确错误。

## 幂等设计

系统在三个入口使用幂等设计：

- 下单幂等：`userId + requestId` 唯一，重复下单返回原订单，不重复扣库存。
- 支付流水创建幂等：同一 `requestId` 返回原支付流水，同一订单已有 `PENDING` 流水时直接复用。
- 支付回调幂等：相同 `callbackRequestId` 直接返回首次处理结果，避免重复改订单、重复触发风控或重复写日志。
- 退款回调幂等：相同 `callbackRequestId` 直接返回首次处理结果，避免重复改退款流水或重复写关键业务日志。

## 支付流水

订单状态描述票务订单处于哪个业务阶段，支付状态描述一次支付请求的渠道处理结果。系统将二者拆分建模：

- 订单：`PENDING_PAYMENT`、`PAID`、`CLOSED`、`REFUNDED`
- 支付流水：`PENDING`、`SUCCESS`、`FAILED`

支付失败只表示本次支付请求失败，不代表订单关闭。订单仍保持待支付，用户可以重新创建支付流水或等待超时关闭。

支付回调额外校验签名、时间戳和金额一致性。`paymentNo` 是系统内部支付流水号，`channelPaymentNo` 是外部渠道返回的支付流水号。

## 退款流水

退票成功后订单立即进入 `REFUNDED`，库存释放并触发退票后风控；资金处理通过退款流水单独跟踪：

- 退款流水：`PENDING`、`SUCCESS`、`FAILED`
- 回调校验：HMAC-SHA256 签名、时间戳容忍窗口、金额一致性
- 回调幂等：使用 `callbackRequestId`

退款回调不会重复释放库存，也不会重复触发退票风控。票务状态和资金退款状态分离后，系统可以清晰处理“票已退、退款处理中”或“票已退、退款失败需补偿”等场景。

## 风控规则引擎

风控逻辑通过 `RiskRule` 接口拆分成独立规则，并通过 `RiskScene` 区分触发场景。当前规则包括：

- 短时间多次购票。
- 单日有效订单金额过高。
- 频繁退票。

支付成功后触发购票相关规则，退票成功后触发退票相关规则，避免待支付订单污染有效交易统计。

## 风险处置闭环

风险事件使用 `RiskStatus` 表示处理结论：

- `PENDING`：待人工审核。
- `CONFIRMED`：确认风险。
- `FALSE_POSITIVE`：误报。
- `CLOSED`：关闭归档。

每次处置都会保存备注、处理人、处理时间，并写入 `risk_event_handle_records`。风险事件表保存当前状态，处置历史表保存状态变化过程。

## 查询缓存

车次余票查询按 `出发站 + 到达站 + 乘车日期` 构建缓存 Key，默认使用本地 TTL 缓存降低重复查询开销，也可以通过配置切换到 Redis。缓存服务保留本地 fallback，避免默认演示环境强依赖 Redis。

下列动作会在事务提交后失效对应线路日期缓存：

- 创建待支付订单。
- 支付成功。
- 关闭待支付订单。
- 超时关闭订单。
- 退票成功。

## 接口限流

系统对高频接口使用固定窗口限流：

- 车次查询：按登录用户或 IP 限流。
- 创建订单：按请求用户 ID 限流。
- 支付回调：按支付流水号和 IP 限流。
- 风险处置：按当前操作人限流。

默认使用本地内存限流，可切换到 Redis `INCR + EXPIRE` 模式。限流阈值集中配置在 `railway.rate-limit.rules`，Controller 只传入规则名和业务 key。超过阈值时返回 429 和 `TOO_MANY_REQUESTS`。

## Outbox 事件

系统新增 `outbox_events` 表，在业务事务内记录订单、支付、退款和风险相关领域事件。事件发布通过 `OutboxEventPublisher` 完成，默认写入 `PENDING` 状态、重试次数和 JSON 载荷。

`OutboxEventDispatcher` 负责扫描待处理事件，将事件标记为 `PROCESSING` 后交给对应 handler。处理成功后标记 `DONE`，处理失败后记录 `lastError` 并增加重试次数；达到最大重试次数后标记为 `FAILED`。

当前阶段保留原同步风控、缓存失效和关键操作日志，Outbox 主要承担事件记录、轻量处理、失败观测、手动重试和统计监控。`FAILED` 事件可由管理员单条或批量重新入队，统计接口返回状态分布、类型分布、失败率和积压数量。后续接入消息队列时，可以把 handler 替换为消息生产者。

## 权限审计

系统使用 Spring Security 作为认证过滤链，登录成功后签发 JWT，密码使用 BCrypt 存储。JWT 过滤器解析 `Authorization: Bearer {token}` 并写入 SecurityContext，敏感接口通过 `@RequiredRole` 控制访问。

- `ADMIN`：可查看日志、处理风险事件、清空缓存。
- `RISK_OFFICER`：可查看日志、处理风险事件。
- `OPERATOR`：可查看运营数据，不能处理风险事件。
- `USER`：普通乘客，只能访问乘客端接口、当前登录用户信息和公开车次查询。

操作日志记录关键业务动作，风险处置历史记录人工处置动作，两者共同支撑审计追踪。

## 双端业务链路

系统新增 `/api/passenger/**` 作为乘客端后端边界。乘客端不创建独立订单表，也不复制状态机，而是复用管理端已经沉淀的订单、支付、退款、风控、Outbox 和审计服务。

乘客下单时，服务端从 JWT 中读取 `userId`，组装为内部 `CreateOrderRequest` 后调用 `OrderService`。乘客支付时先校验订单归属，再复用 `PaymentService` 创建支付流水并走模拟支付成功回调，使支付流水、订单状态、风控触发和 Outbox 事件保持同一链路。乘客退票同样复用 `OrderService.refund`，退票后自动创建退款流水并触发退票后风控。

这种设计保证用户端产生的数据天然进入管理端视图：订单管理、支付退款流水、风险事件、操作日志和 Outbox 事件中心都能看到乘客端操作结果。权限边界由 `USER` 角色和订单归属校验共同保证。

## 电子票与订单详情

电子票使用独立的 `ticket_records` 表建模，不改变订单状态机。支付成功后，`TicketService` 在同一交易链路中为订单签发电子票；退票成功后，电子票状态更新为 `REFUNDED` 并写入失效时间。

订单详情聚合由 `OrderDetailService` 负责。乘客端详情接口会先校验订单归属，只返回订单、电子票、支付流水和退款流水；管理端详情接口额外聚合风险事件、Outbox 事件和最近操作日志，形成单笔交易追踪视图。

## 常用乘车人与实名快照

乘客端通过 `passenger_profiles` 维护常用乘车人。资料归属当前 `USER`，接口层统一校验 JWT 中的 `userId`，避免跨用户读取或使用乘车人资料。

下单时可以传入 `travelerId`。`PassengerService` 会把乘车人资料转换为内部 `CreateOrderRequest`，继续复用 `OrderService` 的库存锁定、订单状态机和 `requestId` 幂等逻辑。订单保存完整交易所需实名字段，同时对外响应只暴露 `passengerIdNoMasked` 和 `passengerPhoneMasked`。

支付成功生成电子票时，`TicketService` 将订单中的实名快照写入 `ticket_records`。因此常用乘车人后续被编辑或删除时，历史订单和电子票仍保留当时购票使用的脱敏快照。

## OpenAPI 与 Docker

系统通过 springdoc-openapi 暴露 `/v3/api-docs` 和 Swagger UI。Swagger 页面允许匿名访问，但业务接口仍按 JWT 和 `@RequiredRole` 校验执行。OpenAPI 配置声明 Bearer Token 安全方案，便于在 Swagger UI 中调试受保护接口。

工程化运行分为两种模式：

- 默认模式：H2、本地缓存、本地限流，用于本地快速启动和 CI 测试。
- `docker` profile：MySQL、Redis 缓存、Redis 限流，用于 Docker Compose 本地联调。

Docker Compose 编排后端、MySQL 和 Redis，演示配置通过环境变量注入 JWT、支付回调和退款回调密钥。

## 演示数据初始化

`DemoDataInitializer` 默认启用，通过 `railway.demo-data.enabled` 控制。初始化器先保证三个演示账号存在，再按模块幂等写入车站、车次、库存、订单、支付流水、退款流水、风险事件、处置历史、操作日志和 Outbox 事件。

演示数据使用固定业务编号和 `DEMO` 前缀，重复启动不会重复插入同一批交易记录。订单数据覆盖待支付、已支付、已关闭和已退票状态；支付、退款、风险和 Outbox 数据也覆盖主要状态，便于看板、分页列表和统计报表首次打开即可展示真实分布。

## 前端管理台

前端保持原生 HTML/CSS/JavaScript，不引入框架或构建工具。页面结构拆分为登录屏、侧边导航、顶部状态栏和内容工作区。登录后通过 JWT Bearer Token 调用受保护接口。

管理台重点展示以下内容：

- 运营看板：订单状态、退票率、风险率、热门车次、缓存和限流指标。
- 交易链路：订单、支付流水和退款流水分页筛选及模拟回调操作。
- 风险运营：风险事件筛选、处置、历史查看和风险报表。
- 事件中心：Outbox 事件分页、状态统计、失败重试和手动派发。
- 审计日志：展示关键业务操作记录。

样式系统使用浅色工作区、深色侧边导航、语义状态色、紧凑表格、指标卡和响应式布局。低库存、风险、失败和待处理状态通过文字和颜色共同表达，避免只依赖颜色传递信息。

## 双端前端接入说明

前端当前拆分为入口页、乘客端和运营管理端。入口页不保存业务状态，只负责引导进入 `passenger.html` 或 `admin.html`。

乘客端使用独立脚本 `passenger.js`，并使用 `railway-passenger-auth` 保存 USER token。这样可以避免普通乘客 token 和管理端 `railway-auth` 状态互相覆盖。乘客端所有订单、支付和退款操作均通过 `/api/passenger/**` 完成，后端继续根据 JWT 中的用户 ID 控制数据边界。

运营管理端迁移到 `admin.html` 后继续使用原 `app.js`，管理端接口语义和 DOM 结构保持不变，避免影响已有运营看板、风险处置、Outbox 和审计日志能力。
# 站内通知中心技术补充

通知中心通过 `NotificationService` 接入现有业务服务：订单创建、支付成功、出票成功、订单关闭、退票、退款成功和退款失败都会生成归属于乘客的站内通知。通知使用 `business_type + business_id + type + user_id` 做幂等去重，并在创建成功后发布 `NOTIFICATION_CREATED` Outbox 事件。当前阶段只提供系统内通知，不引入短信、邮件、外部推送或消息队列。
