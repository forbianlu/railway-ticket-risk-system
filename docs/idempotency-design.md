# 订单幂等提交设计

## 问题背景

购票接口可能因为用户重复点击、浏览器重试、网关重试或网络超时重放而收到多次相同请求。如果每次请求都直接扣库存并创建订单，就会造成重复订单和重复扣票。

## 当前方案

项目使用 `userId + requestId` 作为幂等键。

| 设计点 | 当前实现 |
| --- | --- |
| 幂等字段 | `ticket_orders.request_id` |
| 幂等范围 | 同一用户内唯一 |
| 唯一约束 | `user_id + request_id` |
| 前端生成 | 使用 `crypto.randomUUID()` 生成请求号 |
| 重复请求处理 | 返回第一次创建的订单 |
| 副作用控制 | 重复请求不扣库存、不触发风控、不写重复日志 |

## 下单流程

1. 前端点击购票，生成 `requestId`。
2. 后端收到 `POST /api/orders`。
3. 如果请求携带 `requestId`，先查询是否存在 `userId + requestId` 对应订单。
4. 已存在则直接返回原订单。
5. 不存在则继续校验库存、扣减库存、创建订单。
6. 订单保存时数据库唯一约束保护幂等键不重复。

## 请求示例

```json
{
  "userId": 1001,
  "requestId": "8f7f5c41-b6fd-48ab-8ec5-96b08d3c26d1",
  "trainId": 1,
  "inventoryId": 1,
  "passengerName": "张三",
  "passengerIdCard": "110101200001010011"
}
```

## 已验证场景

集成测试 `shouldReturnExistingOrderForDuplicateRequestId` 会使用同一个 `requestId` 连续提交两次。

断言结果：

- 两次响应的订单 ID 相同
- 两次响应的订单号相同
- 库存只扣减 1 次
- 数据库只新增 1 条订单

## 后续优化

- 为幂等请求保存请求参数摘要，重复请求参数不一致时返回明确错误。
- 将幂等键提升到请求头，例如 `Idempotency-Key`。
- 在网关层对同一用户、同一幂等键做短时间防重。
- 将幂等记录拆成独立表，支持更完整的请求处理中、成功、失败状态。
