# 支付流水与回调幂等设计

## 为什么需要支付流水表

订单状态机解决的是订单侧流转，但真实交易系统还需要记录支付侧处理过程。支付流水表用于保存每一次模拟支付请求，记录支付流水号、金额、渠道、外部渠道流水号、状态、回调请求号和支付成功时间。

这样可以把“订单是否生效”和“支付渠道处理结果”拆开：

- 订单关注库存、乘客、车次和业务状态。
- 支付流水关注支付请求、支付渠道、回调结果和回调幂等。

## 支付流水和订单的关系

当前关系：

- 一个订单可以有多条支付流水。
- 同一订单同一时间只允许存在一条 `PENDING` 支付流水。
- `FAILED` 流水不会改变订单状态，用户可以重新创建新的支付流水。
- `SUCCESS` 流水会将订单从 `PENDING_PAYMENT` 确认为 `PAID`。

## 支付状态和订单状态的区别

| 维度 | 状态 | 说明 |
| --- | --- | --- |
| 支付流水 | `PENDING` | 支付请求已创建，等待模拟回调 |
| 支付流水 | `SUCCESS` | 支付渠道确认成功 |
| 支付流水 | `FAILED` | 支付渠道确认失败 |
| 订单 | `PENDING_PAYMENT` | 订单已创建并锁定库存，等待支付 |
| 订单 | `PAID` | 支付成功，订单生效 |
| 订单 | `CLOSED` | 待支付订单关闭或超时关闭 |
| 订单 | `REFUNDED` | 已支付订单退票 |

支付失败不等于订单关闭。失败后订单仍保持待支付，用户可以重新发起支付，或者等待超时关闭释放库存。

## 支付回调为什么需要幂等

真实支付渠道可能因为网络抖动、超时重试、网关重放等原因，多次发送同一笔支付结果。如果系统每次都执行“订单改为已支付、触发风控、写日志”，就会出现重复风控事件和重复业务日志。

因此支付回调必须幂等。

## callbackRequestId 如何防止重复回调

支付回调请求携带 `callbackRequestId`。系统处理回调前先查询是否已经存在相同 `callbackRequestId` 的支付流水：

- 如果存在，直接返回第一次处理后的结果。
- 如果不存在，再按 `paymentNo` 查询支付流水并处理状态流转。

同时，支付流水已经是 `SUCCESS` 时，再次收到成功回调也直接返回原流水，不重复触发风控。

## 支付回调签名校验

支付回调增加 HMAC-SHA256 签名校验，防止回调参数被篡改或旧回调被重放。签名原文使用固定字段顺序：

```text
paymentNo={paymentNo}&callbackRequestId={callbackRequestId}&amount={amount}&success={success}&timestamp={timestamp}
```

后端使用 `railway.payment.callback-secret` 计算签名，并校验：

- `signature-enabled=true` 时必须携带签名。
- 签名必须与后端计算结果一致。
- `timestamp` 必须在 `timestamp-tolerance-seconds` 容忍窗口内。
- 金额字段参与签名，避免回调金额被篡改。

## 支付金额一致性校验

支付成功或失败回调都必须携带 `amount`。系统使用 `BigDecimal.compareTo` 比较回调金额和支付流水金额，避免 `553.0` 与 `553.00` 这类小数位差异导致误判。

如果金额不一致：

- 支付流水不会变为 `SUCCESS`。
- 订单不会变为 `PAID`。
- 不会触发支付后风控。
- 会记录必要的业务日志用于排查。

## paymentNo 与 channelPaymentNo

- `paymentNo`：系统内部支付流水号，由本系统生成，用于内部查询、幂等和关联订单。
- `channelPaymentNo`：外部支付渠道返回的支付流水号，支付成功回调时必填并保存，用于后续对账。

## 重复回调如何避免重复触发风控

支付成功回调只在支付流水从 `PENDING` 变为 `SUCCESS` 时调用订单支付逻辑。订单支付逻辑会将订单从 `PENDING_PAYMENT` 改为 `PAID`，然后触发风控。

重复回调的处理路径会直接返回已有支付流水，不再调用订单支付逻辑，因此不会重复触发风控。

## 支付失败为什么不释放库存

支付失败只表示本次支付渠道处理失败，并不代表用户放弃订单。订单仍处于 `PENDING_PAYMENT`，库存继续锁定到支付截止时间。

用户可以：

- 重新创建支付流水并再次支付。
- 主动关闭订单释放库存。
- 等待超时关闭任务释放库存。

## 失败流水为什么不能再改成功

当前规则中，`FAILED` 支付流水是终态，不允许同一个 `paymentNo` 再从失败改为成功。如果用户需要重新支付，应创建新的支付流水。

这样设计更容易保证支付记录清晰，也能避免同一条支付流水在失败和成功之间反复翻转导致审计语义混乱。

## 设计价值

这部分设计能体现：

- 交易系统中订单状态和支付状态的边界。
- 支付回调签名、金额校验和幂等的必要性。
- 如何避免重复回调造成重复风控和重复日志。
- 支付失败、订单关闭、退票之间的业务区别。
- 支付流水在交易追踪、异常排查和审计对账中的作用。
