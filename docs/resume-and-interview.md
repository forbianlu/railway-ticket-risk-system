# 简历与面试材料

## 简历项目描述

铁路客运票务与风控运营管理系统  
基于 Spring Boot 开发铁路客运票务与风控运营管理系统，实现车站车次查询、余票扣减、购票退票、异常订单识别、风险事件处置、角色权限、运营数据看板和操作日志审计。使用 JPA 建模核心业务表，通过事务和乐观锁保证订单状态流转与库存一致性，并用并发集成测试验证 16 个请求抢 1 张票时只生成 1 个订单；为车次余票查询设计本地 TTL 缓存，并在购票、退票事务提交后失效线路日期缓存；将风控逻辑抽象为 `RiskRule` 规则引擎，支持短时间多次购票、高金额订单、频繁退票等规则扩展；使用签名令牌、拦截器和 `@RequiredRole` 注解保护风控处置与审计接口，并使用集成测试覆盖核心接口链路。

## 可写技术点

- 使用 Spring Boot 分层架构组织 Controller、Service、Repository、Domain
- 使用 H2 初始化演示数据，后续可平滑切换 MySQL
- 使用事务保证下单扣库存、退票释放库存与订单状态更新一致
- 使用 JPA 乐观锁降低高并发扣库存时的超卖风险
- 使用多线程集成测试验证并发抢票不会产生超卖
- 使用本地 TTL 缓存优化高频车次余票查询，并在库存变更后主动失效缓存
- 使用规则引擎结构识别异常交易行为，规则按触发场景解耦
- 支持风险事件待处理、已处理流转，形成风控运营闭环
- 使用签名令牌、HandlerInterceptor 和 `@RequiredRole` 实现轻量级角色权限控制
- 使用操作日志记录下单、退票、风控触发等关键动作
- 使用集成测试覆盖查询、下单、退票、风控、权限保护和看板接口
- 使用 Docker Compose 和 GitHub Actions 提升工程化完整度

## 面试讲法

### 1. 你为什么做这个项目？

我目标岗位偏铁路局和银行科技岗，所以选择了票务交易和风控这类业务。铁路场景能体现车次、余票、订单、退票等客运业务理解；银行场景能体现交易状态流转、风控规则和日志审计能力。

### 2. 订单创建流程是什么？

接口接收用户、车次和库存 ID 后，服务层先校验库存是否存在、车次和库存是否匹配、余票是否充足。校验通过后扣减库存，生成订单号并创建已支付订单，最后写入操作日志并触发风控规则。

### 3. 为什么要加事务？

下单时涉及库存扣减和订单创建，退票时涉及订单状态更新和库存释放。如果中间任一步失败，数据就会不一致，所以需要事务保证这些数据库操作要么一起成功，要么一起回滚。

### 4. 你怎么防止余票超卖？

第一版使用 JPA 乐观锁，在库存表增加 `version` 字段。多个请求同时扣同一条库存时，只有先提交的事务能成功更新版本号，后提交的请求会触发乐观锁冲突，系统返回“库存正在被其他请求更新，请刷新后重试”。我还写了并发集成测试：动态创建只有 1 张票的库存，使用 16 个线程同时下单，最后断言只有 1 个订单成功、库存为 0、订单数没有超过库存。真实生产里还可以继续结合 Redis 预扣库存、数据库行锁或消息队列削峰。

### 5. 风控规则怎么设计？

我把风控拆成了 `RiskRule` 接口和多个规则实现，每条规则声明自己的触发场景，比如下单后或退票后。当前规则包括同一用户 10 分钟内多次下单、同一用户 7 天内频繁退票、单日有效订单金额过高。命中规则后生成风险事件，风控人员可以标记已处理，处理动作也会进入审计日志。

### 6. 登录权限怎么设计？

我实现的是演示版轻量权限，不是直接引入完整 Spring Security。用户登录后，后端校验账号密码并签发带过期时间的 HMAC-SHA256 签名令牌；前端请求受保护接口时携带 `Authorization: Bearer token`；后端拦截器解析令牌并放入 `AuthContext`，再读取接口上的 `@RequiredRole` 注解判断当前角色是否允许访问。比如风控事件处理只允许 `RISK_OFFICER` 和 `ADMIN`，运营人员调用会返回 403。

### 7. 为什么没有一开始就用 Spring Security？

这个项目第一阶段重点是把业务链路、风控规则和工程结构跑通，所以先用更容易讲清楚的拦截器和签名令牌实现权限主流程。后续企业化版本可以替换成 Spring Security + JWT + BCrypt，登录、鉴权、密码加密和权限表达会更接近生产项目。

### 8. 车次查询缓存怎么设计？

车次余票查询是典型高频读接口，所以我按 `出发站 + 到达站 + 乘车日期` 做缓存 Key，缓存值是车次和座位库存查询结果。查询时先读缓存，命中就直接返回副本；未命中再查数据库并写入缓存。为了避免余票长期不一致，我设置了较短 TTL，同时在购票和退票事务提交后删除对应线路日期的缓存。如果后续部署多台后端，可以把当前本地缓存替换成 Redis，Key 和失效策略保持不变。

### 9. 项目还能怎么优化？

- 将本地车次查询缓存替换为 Redis，支持多实例共享缓存
- 增加并发扣库存压测，并比较乐观锁、行锁、Redis 预扣库存的取舍
- 增加订单幂等号，防止用户重复点击造成重复提交
- 对同一用户、同一库存加短时间接口限流
- 接入 Spring Security + JWT + BCrypt 做生产化权限控制
- 使用 MyBatis-Plus 和 MySQL 贴近企业常见技术栈
- 增加接口测试、压力测试和异常场景测试
