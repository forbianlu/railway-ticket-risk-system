# 简历与面试材料

## 简历项目描述

铁路客运票务与风控运营管理系统  
基于 Spring Boot 开发铁路客运票务与风控运营管理系统，实现车站车次查询、待支付锁票、支付流水、模拟支付回调、超时关闭、退票释放库存、订单分页筛选、异常订单识别、风险事件处置、角色权限、运营数据看板和操作日志审计。使用 JPA 建模核心业务表，通过事务和订单状态机保证 `PENDING_PAYMENT`、`PAID`、`REFUNDED`、`CLOSED` 状态与库存一致；使用乐观锁和并发集成测试验证 16 个请求抢 1 张票时只生成 1 个待支付订单；下单接口支持 `requestId` 幂等提交，重复请求返回原订单且不重复扣库存；支付链路新增 `payment_records` 流水表，使用 `requestId` 防重复创建支付流水、使用 `callbackRequestId` 保证第三方支付回调幂等，重复成功回调不重复修改订单、不重复触发风控；订单列表使用 Spring Data JPA `Specification` + `PageRequest` 支持用户、状态、订单号、创建日期组合筛选；运营看板统计待支付、已支付、已关闭、已退票、退票率、风险率和未处理风险；将风控逻辑抽象为 `RiskRule` 规则引擎，支付成功后触发短时间多次购票、高金额订单规则，退票后触发频繁退票规则；风险事件使用 `PENDING`、`CONFIRMED`、`FALSE_POSITIVE`、`CLOSED` 状态机记录处置结论，并通过处置历史表保存备注、处理人和处理时间；使用签名令牌、拦截器和 `@RequiredRole` 注解保护风控处置与审计接口，并使用集成测试覆盖核心接口链路。

## 可写技术点

- 使用 Spring Boot 分层架构组织 Controller、Service、Repository、Domain
- 使用 H2 初始化演示数据，后续可平滑切换 MySQL
- 使用事务保证下单扣库存、退票释放库存与订单状态更新一致
- 设计订单状态机，支持待支付、已支付、已退票、已关闭等状态流转
- 使用定时任务批量关闭超时待支付订单，释放已锁定库存
- 使用 `requestId` 实现订单幂等提交，避免重复点击或网络重试造成重复扣库存
- 设计 `payment_records` 支付流水表，区分订单状态和支付状态，保留支付号、支付渠道、回调请求号和支付时间
- 使用 `callbackRequestId` 实现支付回调幂等，重复回调直接返回已有流水，避免重复触发风控和重复写支付成功日志
- 使用 JPA 乐观锁降低高并发扣库存时的超卖风险
- 使用多线程集成测试验证并发抢票不会产生超卖
- 使用 Spring Data JPA `Specification` 和 `PageRequest` 实现订单分页与多条件筛选
- 设计运营看板指标，展示订单状态分布、退票率、风险率和未处理风险数量
- 使用本地 TTL 缓存优化高频车次余票查询，并在库存变更后主动失效缓存
- 使用规则引擎结构识别异常交易行为，规则按触发场景解耦
- 设计风险事件状态机，支持待处理、确认风险、误报、关闭归档等处置结论
- 设计风险处置历史表，保存每次处置的前后状态、备注、处理人和处理时间
- 使用签名令牌、HandlerInterceptor 和 `@RequiredRole` 实现轻量级角色权限控制
- 使用操作日志记录下单、支付、关闭、退票、风控触发等关键动作
- 使用集成测试覆盖查询、下单、支付流水、支付回调幂等、关闭、退票、风控、权限保护、缓存失效和看板接口
- 使用 Docker Compose 和 GitHub Actions 提升工程化完整度

## 面试讲法

### 1. 你为什么做这个项目？

我目标岗位偏铁路局和银行科技岗，所以选择了票务交易和风控这类业务。铁路场景能体现车次、余票、订单、退票等客运业务理解；银行场景能体现交易状态流转、风控规则和日志审计能力。

### 2. 订单创建流程是什么？

接口接收用户、车次和库存 ID 后，服务层先校验幂等号，再校验库存是否存在、车次和库存是否匹配、余票是否充足。校验通过后扣减库存，生成订单号并创建 `PENDING_PAYMENT` 待支付订单，同时设置 15 分钟支付截止时间。用户调用支付接口后订单变为 `PAID`，写入支付日志并触发下单后风控规则；如果待支付订单超时或主动关闭，系统会把订单变为 `CLOSED` 并释放库存。

### 3. 为什么要加事务？

下单时涉及库存扣减和订单创建，支付时涉及订单状态更新和风控生成，关闭或退票时涉及订单状态更新和库存释放。如果中间任一步失败，数据就会不一致，所以需要事务保证这些数据库操作要么一起成功，要么一起回滚。

### 4. 你怎么防止余票超卖？

第一版使用 JPA 乐观锁，在库存表增加 `version` 字段。多个请求同时锁定同一条库存时，只有先提交的事务能成功更新版本号，后提交的请求会触发乐观锁冲突，系统返回“库存正在被其他请求更新，请刷新后重试”。我还写了并发集成测试：动态创建只有 1 张票的库存，使用 16 个线程同时下单，最后断言只有 1 个待支付订单成功、库存为 0、订单数没有超过库存。真实生产里还可以继续结合 Redis 预扣库存、数据库行锁或消息队列削峰。

### 5. 风控规则怎么设计？

我把风控拆成了 `RiskRule` 接口和多个规则实现，每条规则声明自己的触发场景，比如支付成功后或退票后。当前规则包括同一用户 10 分钟内多次支付成功、同一用户 7 天内频繁退票、单日有效订单金额过高。命中规则后生成 `PENDING` 风险事件，风控人员可以确认为风险、标记误报或关闭归档，每次处置都会保存备注、处理人、处理时间，并写入处置历史和审计日志。

### 6. 订单幂等怎么设计？

下单请求可以携带 `requestId`，后端用 `userId + requestId` 作为幂等键，并在订单表上加唯一约束。服务层创建订单前先查询这个幂等键，如果已经存在订单，就直接返回原订单，不再扣库存、不再写重复日志。这样用户重复点击或客户端网络重试时，不会生成多笔订单。前端每次购票会自动生成一个 UUID 作为 `requestId`。

### 7. 登录权限怎么设计？

我实现的是演示版轻量权限，不是直接引入完整 Spring Security。用户登录后，后端校验账号密码并签发带过期时间的 HMAC-SHA256 签名令牌；前端请求受保护接口时携带 `Authorization: Bearer token`；后端拦截器解析令牌并放入 `AuthContext`，再读取接口上的 `@RequiredRole` 注解判断当前角色是否允许访问。比如风控事件处理只允许 `RISK_OFFICER` 和 `ADMIN`，运营人员调用会返回 403。

### 8. 为什么没有一开始就用 Spring Security？

这个项目第一阶段重点是把业务链路、风控规则和工程结构跑通，所以先用更容易讲清楚的拦截器和签名令牌实现权限主流程。后续企业化版本可以替换成 Spring Security + JWT + BCrypt，登录、鉴权、密码加密和权限表达会更接近生产项目。

### 9. 车次查询缓存怎么设计？

车次余票查询是典型高频读接口，所以我按 `出发站 + 到达站 + 乘车日期` 做缓存 Key，缓存值是车次和座位库存查询结果。查询时先读缓存，命中就直接返回副本；未命中再查数据库并写入缓存。为了避免余票长期不一致，我设置了较短 TTL，同时在锁票、支付、关闭待支付订单和退票事务提交后删除对应线路日期的缓存。如果后续部署多台后端，可以把当前本地缓存替换成 Redis，Key 和失效策略保持不变。

### 10. 项目还能怎么优化？

- 将本地车次查询缓存替换为 Redis，支持多实例共享缓存
- 增加并发扣库存压测，并比较乐观锁、行锁、Redis 预扣库存的取舍
- 对同一用户、同一库存加短时间接口限流
- 接入 Spring Security + JWT + BCrypt 做生产化权限控制
- 使用 MyBatis-Plus 和 MySQL 贴近企业常见技术栈
- 增加接口测试、压力测试和异常场景测试

### 11. 为什么状态机之后要做订单分页筛选和看板指标？

订单状态机解决的是交易链路正确性，但后台运营系统还需要“查得出来、看得明白”。订单列表如果只能返回最近几条记录，面试时容易被认为只是演示接口；加入用户、状态、订单号、日期和分页后，更接近真实管理台的查询能力。看板增加待支付、已关闭、退票率、风险率后，可以从运营视角解释订单积压、支付转化、退票情况和风险压力，这比只展示总订单数更有业务含义。

### 12. 订单分页筛选怎么实现？

后端用 `JpaSpecificationExecutor` 动态组合查询条件：有 `userId` 就加用户条件，有 `status` 就加状态条件，有 `fromDate` 和 `toDate` 就按创建时间范围过滤，有 `orderNo` 就做订单号模糊查询。分页使用 `PageRequest`，按 `createdAt` 和 `id` 倒序排序，接口返回 `content`、`page`、`size`、`totalElements`、`totalPages`、`first`、`last`。这样实现没有引入复杂依赖，也便于通过集成测试验证。

### 13. 看板指标怎么计算？

看板保留原有总订单、有效订单、退票订单、未处理风险和热门车次，同时新增状态统计：待支付、已支付、已关闭、已退票。退票率按 `refundedOrderCount / max(1, paidOrderCount + refundedOrderCount)` 计算，风险率按 `totalRiskEvents / max(1, paidOrderCount + refundedOrderCount)` 计算，使用 `max(1, ...)` 避免没有有效订单时出现除零错误。

### 14. 为什么要增加支付流水和回调幂等？

订单状态机只能说明订单从待支付到已支付的业务结果，但银行和支付系统更关注交易过程本身是否可追踪、可重试、可对账。所以我单独增加了 `payment_records` 支付流水表，记录支付流水号、订单号、用户、金额、渠道、支付状态、创建时间、支付成功时间和回调请求号。订单状态回答“票务订单现在处于什么业务阶段”，支付状态回答“这一次支付请求是否成功”，两者分开后更接近真实交易系统。

支付回调可能因为网络超时被第三方重复推送，所以接口使用 `callbackRequestId` 做幂等。服务层先根据回调请求号查询是否处理过，如果已经处理过就直接返回第一次的支付流水；如果支付流水已经是 `SUCCESS`，再次收到成功回调也只返回已有结果，不再调用订单支付逻辑。这样可以避免重复把订单改为已支付、重复触发风控规则、重复写支付成功审计日志。支付失败时流水变为 `FAILED`，订单仍保持 `PENDING_PAYMENT`，库存继续锁定，用户可以重新创建支付流水或等待订单超时关闭。

### 15. 为什么风控事件不能只用 handled 布尔值？

`handled=true/false` 只能回答“有没有处理”，但不能说明处理结论是什么。真实风控运营里，人工审核可能确认风险，也可能判断为误报，还可能先确认再关闭归档。如果只用布尔值，面试时很难讲清楚风险事件从规则命中到人工审核、复核归档的完整闭环。所以我把风险事件升级为 `RiskStatus` 状态机：`PENDING` 表示待处理，`CONFIRMED` 表示确认风险，`FALSE_POSITIVE` 表示误报，`CLOSED` 表示归档关闭；同时保留 `handled` 字段兼容旧前端，当前规则是 `status != PENDING` 时视为已处理。

处置历史表的作用是保留“谁在什么时候把事件从什么状态改成了什么状态，以及备注是什么”。这比只在风险事件表里保存最新状态更适合审计，也更贴近银行、铁路和央国企后台系统对可追溯性的要求。权限上只有 `ADMIN` 和 `RISK_OFFICER` 能处置，`OPERATOR` 只能查看运营数据不能改风控结论；每次处置还会写入操作日志，形成“状态、历史、权限、审计”四件套。
