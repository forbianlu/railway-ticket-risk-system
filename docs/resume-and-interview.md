# 简历与面试材料

## 简历项目描述

铁路客运票务与风控运营管理系统  
基于 Spring Boot 开发铁路客运票务与风控运营管理系统，实现车站车次查询、余票扣减、购票退票、异常订单识别、风险事件处置、运营数据看板和操作日志审计。使用 JPA 建模核心业务表，通过事务和乐观锁保证订单状态流转与库存一致性；将风控逻辑抽象为 `RiskRule` 规则引擎，支持短时间多次购票、高金额订单、频繁退票等规则扩展，并使用集成测试覆盖核心接口链路。

## 可写技术点

- 使用 Spring Boot 分层架构组织 Controller、Service、Repository、Domain
- 使用 H2 初始化演示数据，后续可平滑切换 MySQL
- 使用事务保证下单扣库存、退票释放库存与订单状态更新一致
- 使用 JPA 乐观锁降低高并发扣库存时的超卖风险
- 使用规则引擎结构识别异常交易行为，规则按触发场景解耦
- 支持风险事件待处理、已处理流转，形成风控运营闭环
- 使用操作日志记录下单、退票、风控触发等关键动作
- 使用集成测试覆盖查询、下单、退票、风控和看板接口
- 使用 Docker Compose 和 GitHub Actions 提升工程化完整度

## 面试讲法

### 1. 你为什么做这个项目？

我目标岗位偏铁路局和银行科技岗，所以选择了票务交易和风控这类业务。铁路场景能体现车次、余票、订单、退票等客运业务理解；银行场景能体现交易状态流转、风控规则和日志审计能力。

### 2. 订单创建流程是什么？

接口接收用户、车次和库存 ID 后，服务层先校验库存是否存在、车次和库存是否匹配、余票是否充足。校验通过后扣减库存，生成订单号并创建已支付订单，最后写入操作日志并触发风控规则。

### 3. 为什么要加事务？

下单时涉及库存扣减和订单创建，退票时涉及订单状态更新和库存释放。如果中间任一步失败，数据就会不一致，所以需要事务保证这些数据库操作要么一起成功，要么一起回滚。

### 4. 你怎么防止余票超卖？

第一版使用 JPA 乐观锁，在库存表增加 `version` 字段。多个请求同时扣同一条库存时，只有先提交的事务能成功更新版本号，后提交的请求会触发乐观锁冲突，系统返回“库存正在被其他请求更新，请刷新后重试”。真实生产里还可以继续结合 Redis 预扣库存、数据库行锁或消息队列削峰。

### 5. 风控规则怎么设计？

我把风控拆成了 `RiskRule` 接口和多个规则实现，每条规则声明自己的触发场景，比如下单后或退票后。当前规则包括同一用户 10 分钟内多次下单、同一用户 7 天内频繁退票、单日有效订单金额过高。命中规则后生成风险事件，风控人员可以标记已处理，处理动作也会进入审计日志。

### 6. 项目还能怎么优化？

- Redis 缓存热门车次余票，降低数据库查询压力
- 使用乐观锁或数据库行锁处理并发扣库存
- 接入 Spring Security + JWT 做权限控制
- 使用 MyBatis-Plus 和 MySQL 贴近企业常见技术栈
- 增加接口测试、压力测试和异常场景测试
