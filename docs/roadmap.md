# 后续开发路线

## 阶段一：运行与验证

- 安装 Maven。
- 启动 Spring Boot 后端。
- 打开前端页面完成一次查车、下单、支付、退票。
- 检查 H2 控制台中的订单、库存、支付流水、退款流水和风险事件数据。
- 检查默认演示数据在看板、订单、支付、退款、风险和 Outbox 页面中的展示效果。
- 执行 Maven 集成测试和前端脚本语法检查。

## 阶段二：基础能力增强

- 完善接口异常场景测试。
- 增加更多订单状态非法流转用例。
- 增加支付失败、退款失败、重复回调和重复下单的边界测试。
- 完成入口选择页和乘客购票前端，接入 `/api/passenger/**` 展示查票、下单、支付、退票和我的订单。
- 增强乘客端常用乘车人资料和实名快照展示，支持下单时选择常用乘车人并在订单、电子票中保留脱敏历史快照。
- 平台化整理 README，将乘客端和运营管理端统一描述为双端系统。
- 继续补充 H2 和 Docker MySQL 模式下的演示数据集，例如节假日高峰和异常退款批次。
- 完善接口文档和数据库字段说明。

## 阶段三：工程化增强

- 补充真实 Redis 环境端到端联调记录、缓存监控和限流指标看板。
- 增加 refresh token、登录失败限制和令牌失效机制。
- 增加接口压测和并发购票验证脚本。
- 配置更完整的 GitHub Actions 自动测试流程。
- 增加 Docker Compose 健康检查、镜像标签和独立部署环境验证。
- 将 Outbox 派发器切换为消息队列生产者。

## 阶段四：运营能力增强

- 为风险事件增加风险等级筛选和处理人筛选。
- 增加风险处理 SLA 和超时提醒。
- 增加订单、支付流水和风险事件导出。
- 增加风险趋势图和运营日报。
- 扩展风险规则配置化能力。
- 增加退款重试、退款人工补偿和支付退款对账报表。
- 增加 Outbox 处理器级别耗时趋势、失败告警和消息队列生产者适配。
# 站内通知中心路线补充

- 后续可在数据量增长时扩展独立索引服务，但当前阶段不引入外部搜索中间件。
