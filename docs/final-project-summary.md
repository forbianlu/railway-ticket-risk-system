# 项目最终总结

## 1. 系统定位

铁路客运票务与风控运营管理系统围绕铁路客运售票、订单支付、库存一致性、风险识别、风险处置、权限控制和运营统计构建。系统模拟客运售票后台中的核心交易链路，并提供运营管理和风险处置能力。

当前实现重点在于后端业务建模、状态流转控制、事务一致性、幂等处理、风控规则、审计追踪和集成测试。

## 2. 当前完成的功能模块

- 车站和车次查询：支持按出发站、到达站、乘车日期查询余票。
- 座位库存：维护车次、日期、座位类型、票价、总库存和剩余库存。
- 订单管理：创建待支付订单、支付、关闭、超时关闭、退票。
- 订单分页筛选：支持按用户、状态、订单号和创建日期查询。
- 支付流水：支持创建模拟支付流水、成功回调、失败回调和流水分页查询。
- 风控规则：支持短时间多次购票、高金额订单、频繁退票规则。
- 风险处置：支持风险状态流转、处置备注、处理人、处理时间和处置历史。
- 风险运营报表：支持风险状态分布、场景分布、误报率和处置完成率统计。
- 查询缓存：车次余票查询支持 local / Redis TTL 缓存和事务后缓存失效。
- 接口限流：关键接口支持本地 / Redis 固定窗口限流。
- 权限审计：支持登录令牌、角色校验、操作日志和风险处置历史。
- 前端管理台：展示运营看板、订单、支付流水和风险事件。
- 集成测试：覆盖核心交易链路、幂等、并发防超卖、缓存、权限、支付回调和风险运营能力。

## 3. 核心技术亮点

- Spring Boot 分层架构：Controller、Service、Repository、Domain、DTO。
- JPA 实体建模：订单、库存、支付流水、风险事件、处置历史和操作日志。
- 事务一致性：下单扣库存、关闭释放库存、退票释放库存、支付触发风控保持事务边界。
- JPA 乐观锁：库存表使用 `version` 防止并发扣减导致超卖。
- 订单状态机：限制 `PENDING_PAYMENT`、`PAID`、`REFUNDED`、`CLOSED` 的合法流转。
- 幂等设计：下单 `requestId`、支付流水创建 `requestId`、支付回调 `callbackRequestId`。
- 风控规则引擎：使用 `RiskRule` + `RiskScene` 让规则按场景解耦。
- 后台查询能力：订单、支付流水、风险事件均支持分页和条件筛选。
- 审计追踪：操作日志和风险处置历史保留关键业务动作。

## 4. 关键业务流程

购票主流程：

```text
查询车次 -> 创建待支付订单 -> 锁定库存 -> 创建支付流水 -> 支付成功回调 -> 订单变为 PAID -> 触发风控
```

关闭流程：

```text
PENDING_PAYMENT -> 手动关闭或超时关闭 -> CLOSED -> 释放库存
```

退票流程：

```text
PAID -> REFUNDED -> 释放库存 -> 触发退票后风控
```

风险处置流程：

```text
支付成功或退票 -> 规则命中 -> 生成 PENDING 风险事件 -> 人工处置 -> 写处置历史 -> 写操作日志
```

## 5. 数据一致性设计

- 创建订单时先校验库存，再扣减库存并创建 `PENDING_PAYMENT` 订单。
- 支付成功只确认订单生效，不重复扣减库存。
- 关闭待支付订单和退票都会释放库存，但业务语义不同。
- 退票只能作用于 `PAID` 订单，关闭只能作用于 `PENDING_PAYMENT` 订单。
- 相关写操作使用事务包裹，库存表使用乐观锁降低并发冲突风险。
- 查询缓存只缓存车次余票读结果，库存相关交易动作提交后主动失效缓存，默认本地模式可切换为 Redis 模式。

## 6. 幂等设计

- 下单幂等：`userId + requestId` 唯一，重复请求返回原订单，不重复扣库存。
- 支付流水创建幂等：同一 `requestId` 返回原支付流水，同一订单已有 `PENDING` 流水时复用。
- 支付回调幂等：相同 `callbackRequestId` 直接返回首次处理结果。
- 重复成功回调不会重复修改订单、不会重复触发风控、不会重复写支付成功业务日志。

## 7. 风控设计

当前风控规则：

- `RapidPurchaseRiskRule`：同一用户 10 分钟内支付成功订单达到 3 次。
- `HighAmountRiskRule`：同一用户当日有效支付金额超过 1000 元。
- `FrequentRefundRiskRule`：同一用户 7 天内退票达到 3 次。

风险状态：

- `PENDING`：待人工审核。
- `CONFIRMED`：确认风险。
- `FALSE_POSITIVE`：误报。
- `CLOSED`：关闭归档。

风险运营报表支持统计风险总量、状态分布、场景分布、确认风险占比、误报率、处置完成率和平均首次处置耗时。

## 8. 权限和审计设计

- 登录后签发带过期时间和签名的演示令牌。
- 后端拦截器解析 `Authorization: Bearer {token}` 并写入认证上下文。
- `@RequiredRole` 控制敏感接口访问。
- `ADMIN` 和 `RISK_OFFICER` 可以处置风险、查看日志。
- `OPERATOR` 可以查看运营数据，但不能处置风险事件。
- 操作日志记录下单、支付、关闭、退票、风险触发、风险处置等关键动作。
- 风险处置历史记录每次状态流转、备注、操作人和操作时间。

## 10. 技术决策说明

- 当前缓存和限流均支持本地模式与 Redis 模式；默认本地模式适合单实例演示，Redis 模式适合多实例共享状态。
- 当前认证使用 Spring Security + JWT + BCrypt，保持无状态访问控制；后续可增加 refresh token、登录失败限制和令牌失效机制。
- 当前支付为模拟支付，重点展示支付流水、回调状态和幂等处理；接入真实渠道时需要增加签名验签和回调来源校验。
- 当前风险规则为同步规则引擎，适合交易后实时生成风险事件；后续可扩展为异步规则计算和风险评分。
