# 项目最终总结

## 1. 项目定位

铁路客运票务与风控运营管理系统是一个面向铁路局信息技术岗、银行科技岗和央国企软件开发岗的后端主导型项目。项目围绕“交易系统 + 风控运营后台”展开，覆盖车次查询、库存锁定、订单状态机、模拟支付、支付回调幂等、风险识别、风险处置、权限审计和运营报表。

项目不依赖 Redis、MQ、Spring Security 或真实支付 SDK，当前版本重点展示 Java 后端业务建模、事务一致性、接口设计、集成测试和文档交付能力。

## 2. 当前完成的功能模块

- 车站和车次查询：支持按出发站、到达站、乘车日期查询余票。
- 座位库存：维护车次、日期、座位类型、票价、总库存和剩余库存。
- 订单管理：创建待支付订单、支付、关闭、超时关闭、退票。
- 订单分页筛选：支持按用户、状态、订单号和创建日期查询。
- 支付流水：支持创建模拟支付流水、成功回调、失败回调和流水分页查询。
- 风控规则：支持短时间多次购票、高金额订单、频繁退票规则。
- 风险处置：支持风险状态流转、处置备注、处理人、处理时间和处置历史。
- 风险运营报表：支持风险状态分布、场景分布、误报率和处置完成率统计。
- 查询缓存：车次余票查询支持本地 TTL 缓存和事务后缓存失效。
- 权限审计：支持登录令牌、角色校验、操作日志和风控处置权限控制。
- 前端管理台：原生 HTML/CSS/JavaScript 展示运营看板、订单、支付流水和风险事件。
- 集成测试：覆盖核心交易链路、幂等、并发防超卖、缓存、权限、支付回调和风险运营能力。

## 3. 核心技术亮点

- Spring Boot 分层架构：Controller、Service、Repository、Domain、DTO。
- JPA 实体建模：订单、库存、支付流水、风险事件、处置历史和操作日志。
- 事务一致性：下单扣库存、关闭释放库存、退票释放库存、支付触发风控保持事务边界。
- JPA 乐观锁：库存表使用 `version` 防止并发扣减导致超卖。
- 订单状态机：限制 `PENDING_PAYMENT`、`PAID`、`REFUNDED`、`CLOSED` 的合法流转。
- 幂等设计：下单 `requestId`、支付流水创建 `requestId`、支付回调 `callbackRequestId`。
- 风控规则引擎：使用 `RiskRule` + `RiskScene` 让规则按场景解耦。
- 后台查询能力：订单、支付流水、风险事件均支持分页和条件筛选。
- 审计追踪：操作日志和风险处置历史保留关键业务动作。

## 4. 关键业务流程

购票主流程：

```text
查询车次 -> 创建待支付订单 -> 锁定库存 -> 创建支付流水 -> 支付成功回调 -> 订单变为 PAID -> 触发风控
```

关闭流程：

```text
PENDING_PAYMENT -> 手动关闭或超时关闭 -> CLOSED -> 释放库存
```

退票流程：

```text
PAID -> REFUNDED -> 释放库存 -> 触发退票后风控
```

风险处置流程：

```text
支付成功或退票 -> 规则命中 -> 生成 PENDING 风险事件 -> 风控人员处置 -> 写处置历史 -> 写操作日志
```

## 5. 数据一致性设计

- 创建订单时先校验库存，再扣减库存并创建 `PENDING_PAYMENT` 订单。
- 支付成功只确认订单生效，不重复扣减库存。
- 关闭待支付订单和退票都会释放库存，但业务语义不同。
- 退款只能作用于 `PAID` 订单，关闭只能作用于 `PENDING_PAYMENT` 订单。
- 相关写操作使用事务包裹，库存表使用乐观锁降低并发冲突风险。
- 查询缓存只缓存车次余票读结果，库存相关交易动作提交后主动失效缓存。

## 6. 幂等设计

- 下单幂等：`userId + requestId` 唯一，重复请求返回原订单，不重复扣库存。
- 支付流水创建幂等：同一 `requestId` 返回原支付流水，同一订单已有 `PENDING` 流水时复用。
- 支付回调幂等：相同 `callbackRequestId` 直接返回首次处理结果。
- 重复成功回调不会重复修改订单、不会重复触发风控、不会重复写支付成功业务日志。

## 7. 风控设计

当前风控规则：

- `RapidPurchaseRiskRule`：同一用户 10 分钟内支付成功订单达到 3 次。
- `HighAmountRiskRule`：同一用户当日有效支付金额超过 1000 元。
- `FrequentRefundRiskRule`：同一用户 7 天内退票达到 3 次。

风险状态：

- `PENDING`：待人工审核。
- `CONFIRMED`：确认风险。
- `FALSE_POSITIVE`：误报。
- `CLOSED`：关闭归档。

风险运营报表支持统计风险总量、状态分布、场景分布、确认风险占比、误报率、处置完成率和平均首次处置耗时。

## 8. 权限和审计设计

- 登录后签发带过期时间和签名的演示令牌。
- 后端拦截器解析 `Authorization: Bearer {token}` 并写入认证上下文。
- `@RequiredRole` 控制敏感接口访问。
- `ADMIN` 和 `RISK_OFFICER` 可以处置风险、查看日志。
- `OPERATOR` 可以查看运营数据，但不能处置风险事件。
- 操作日志记录下单、支付、关闭、退票、风险触发、风险处置等关键动作。
- 风险处置历史记录每次状态流转、备注、操作人和操作时间。

## 10. 简历表述

铁路客运票务与风控运营管理系统：基于 Spring Boot、Spring Data JPA 和 H2/MySQL profile 实现后端主导型交易与风控运营系统，覆盖车次查询、待支付锁票、订单状态机、支付流水、支付回调幂等、退票释放库存、查询缓存、风险规则引擎、风险处置闭环、角色权限、操作日志和运营报表。使用 JPA 乐观锁和集成测试验证并发抢票不超卖，使用 `requestId` 和 `callbackRequestId` 保证下单及支付回调幂等，通过 `RiskRule` 规则接口和 `RiskStatus` 状态机支撑异常交易识别、人工处置和审计追踪。

## 11. 面试问答提纲

- 为什么订单创建后不是直接成功，而是进入 `PENDING_PAYMENT`？
- 如何保证下单扣库存和订单创建的一致性？
- JPA 乐观锁如何防止并发超卖？
- `requestId` 如何避免重复下单？
- 为什么需要支付流水表，支付状态和订单状态有什么区别？
- 支付回调为什么必须幂等，重复回调如何处理？
- 风控规则为什么要拆成 `RiskRule`？
- 为什么风险事件不能只用 `handled` 布尔值？
- 风险处置历史表解决了什么审计问题？
- 为什么后台系统需要订单分页、风险分页和运营报表？
- 当前轻量权限和 Spring Security 的区别是什么？
- 如果生产化，Redis、MQ、JWT、真实支付 SDK 分别会加在哪里？

## 12. 后续扩展方向

- 将本地 TTL 缓存替换为 Redis，支持多实例共享缓存。
- 使用延时队列或定时任务优化超时订单关闭。
- 接入真实支付渠道回调签名校验。
- 将演示令牌替换为 Spring Security + JWT + BCrypt。
- 增加风险等级、处理人、SLA 和导出报表。
- 增加接口压测、异常注入和更细粒度的端到端测试。
